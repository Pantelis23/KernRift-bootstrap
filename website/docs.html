1. Types & Variables

KernRift targets bare-metal x86-64 and ARM64. The type precedes the name (C-style). Every type maps directly to a machine-width storage cell — there are no hidden allocations.

Scalar types

TypeAliasesWidthDescription
uint8u8, byte1 BUnsigned byte.
uint16u162 BUnsigned 16-bit integer.
uint32u324 BUnsigned 32-bit integer.
uint64u64, addr8 BUnsigned 64-bit, pointer-sized.
int8i81 BSigned byte.
int16i162 BSigned 16-bit integer.
int32i324 BSigned 32-bit integer.
int64i648 BSigned 64-bit integer.
float32f324 BIEEE-754 single-precision float.
float64f648 BIEEE-754 double-precision float.
bool—1 BBoolean. Literals: true / false.
char—1 BASCII character. Literal: 'A'.

Slice type

[T] is a fat pointer — a (ptr: uint64, len: uint64) pair under SysV ABI. Slices are views over caller-owned memory; no allocation occurs. Access the element count inside a function as slice.len.

Variable declaration

The type precedes the name. An initialiser is required for most types.

2. Operators

Expressions use a Pratt parser. Precedence from highest (tightest) to lowest:

Prec.OperatorsNotes
110!  ~  - (prefix)Logical not, bitwise not, negation
100*  /  % V2Multiply, divide, remainder
90+  -Add, subtract

70<  >  <=  >=Comparison
60==  !=Equality

Literals: decimal 42, hexadecimal 0x1000, binary 0b1010, float 3.14, char 'A', bool true / false. Use parentheses to override precedence.

uint32 x  = (a + b) & 0xFF
bool   ok = (flags & 0x01) != 0

3. Functions

Functions are the primary unit of computation. Parameters are written as TYPE name (type before name). The return type follows ->; omit it for void functions.

fn get_status() -> uint32 {
    uint32 s = UART0.Status
    return s
}

fn send(uint8 b) {
    UART0.Data = b
}

Parameters

Scalar and slice parameters. Parameters are available as normal local variables throughout the function body, including in comparisons and arithmetic.

fn clamp(uint32 x, uint32 limit) -> uint32 {
    if x > limit {
        return limit
    }
    return x
}

fn write_buf([uint8] data) {
    uint64 n = data.len
}

Calling functions

Functions defined anywhere in the same file can be called by name. Return values are assigned to typed variables. Note (KR0.1): recursive calls — including mutual recursion — are rejected at compile time. Use explicit loops instead.

fn double(uint32 n) -> uint32 {
    uint32 r = n
    r += n
    return r
}

fn quadruple(uint32 n) -> uint32 {
    uint32 r = double(n)
    r = double(r)
    return r
}

@ctx(thread, boot)
fn entry() {
    uint32 x = quadruple(3)   // x == 12
}

Return

Every code path that produces a value must end with an explicit return expr. Void functions may use a bare return or simply fall off the end.

fn get_val() -> uint32 {
    uint32 x = 42
    return x
}

fn maybe_send(uint8 b, bool ok) {
    if !ok {
        return
    }
    UART0.Data = b
}

4. Output (print)

print("...") is a built-in statement that writes a string literal to the UART output buffer. It is the primary output mechanism in the freestanding environment. The argument must be a string literal — variables and expressions are not accepted.

@ctx(thread, boot)
fn entry() {
    print("Booting KernRift...\n")
    print("Init complete.\n")
}

At runtime the kernrift loader maps a 4 KiB UART buffer at 0x10000000. Each print call appends to that buffer via a cursor stored at offset 0. After the entry function returns, kernrift flushes the buffer to stdout.

Escape sequences supported inside string literals:

SequenceMeaning
\nNewline (LF)
\tHorizontal tab
\\Literal backslash
\"Literal double-quote

5. Control Flow

if / else

Conditions are full expressions. Parentheses are optional. The else branch is optional.

if status == 0 {
    uint32 x = 1
} else {
    uint32 x = 2
}

Comparing a function parameter works identically to comparing a local variable:

fn check(uint32 code) {
    if code == 0xFF {
        print("error\n")
    }
}

while

Executes the body as long as the condition is true. The loop variable retains its declared type — uint32 arithmetic stays 32-bit throughout the loop.

uint32 i = 0
while i < n {
    i = i + 1
}

for (range loop)

Exclusive range .. and inclusive range ..=. The loop variable i is an implicit uint64 induction variable — read-only inside the body.

// Exclusive range: 0, 1, ..., n-1
for i in 0..10 {
    total += 1
}

// Inclusive range: 0, 1, ..., 255
for i in 0..=255 {
    sum += i
}

break and continue

while true {
    if done { break }
    if skip { continue }
    total += 1
}

6. Constants

Integer constants only. Constants are inlined at every use site — no runtime storage is allocated.

const uint32 UART_ENABLE = 0x0001
const uint16 BAUD_115200 = 0x001A
const uint64 BASE_ADDR   = 0xFEB00000

7. Structs and Enums

Structs and enums are compile-time tools for address arithmetic. No runtime objects are created.

Structs

Layout is C-style packed: no alignment padding, fields in declaration order. Field offsets are compile-time constants available as NAME::field.

struct UartRegs {
    Control: uint32,   // offset 0
    Status:  uint32,   // offset 4
    Data:    uint8,    // offset 8
}

Enums

Each variant is a named integer constant. Referenced as NAME::VARIANT.

enum UartBase: uint64 {
    Uart0 = 0x40000000,
    Uart1 = 0x40001000,
}

8. Hardware Access (MMIO)

To safely interface with hardware, use the device block. This maps physical registers to typed, volatile-safe memory accesses. Requires the @module_caps(Mmio) capability.

Device declaration

@module_caps(Mmio)

device UART0 at 0x3F000000 {
    Data   at 0x00 : uint8  rw
    Status at 0x04 : uint32 ro
    CR     at 0x08 : uint16 rw
}

Access modes: rw (read-write), ro (read-only), wo (write-only). The compiler enforces these — writing to a ro field is a compile error.

Reading and writing device registers

fn read_status() -> uint32 {
    // Compiler emits a volatile load instruction
    uint32 s = UART0.Status
    return s
}

fn send_byte(uint8 b) {
    UART0.Data = b    // Compiler emits a volatile store
}

Raw MMIO access

For hardware not described by a device block. Requires @module_caps(MmioRaw).

@module_caps(MmioRaw)

fn probe() {
    raw_write<uint32>(0x1014, x)
    raw_read<uint32>(0x1014, result)
}

9. Kernel Safety Annotations

Annotations appear immediately before fn declarations. Multiple annotations can share a line or occupy separate lines; order does not matter. Invalid call edges — for example, calling an allocator from an interrupt handler — are rejected at compile time.

Execution context: @ctx(...)

Declares which execution contexts the function may run in. For any call edge caller → callee, the compiler requires ctx(caller) ⊆ ctx(callee).

ContextMeaning
threadNormal thread / task context.
bootEarly boot, before scheduling is active.
irqHardware interrupt handler.
nmiNon-maskable interrupt (strictest).

Side effects: @eff(...)

EffectMeaning
MmioPerforms memory-mapped I/O.
allocAllocates memory (forbidden in irq).
blockMay put the thread to sleep.
yieldMay voluntarily yield the CPU.

Module capabilities: @module_caps(...)

Top-level declaration listing which capability classes the module may use. The compiler rejects any operation that requires an unlisted capability.

CapabilityMeaning
MmioMMIO via declared device blocks.
MmioRawRaw MMIO at arbitrary addresses.
Ioportx86 I/O port instructions.

Other annotations

// Verified to never yield or block — required on IRQ handlers
@noyield
@ctx(irq)
fn irq_handler() { }

// Aligns entry to 16-byte boundary for I-cache performance
@hotpath
@ctx(irq)
fn fast_dispatch() { }

// Max non-leaf calls between any acquire/release pair = 3
@lock_budget(3)
@ctx(thread)
fn do_work() { }

// Leaf: does not call any other KernRift function
@leaf
fn tiny() { }

10. Locks

Locks are declared at module level. Acquire and release must be balanced on all paths. The compiler builds a lock graph and rejects any cycle that would cause a deadlock.

lock ConsoleLock

fn write_char(uint8 b) {
    acquire(ConsoleLock)
    UART0.Data = b
    release(ConsoleLock)
}

Always acquire locks in a consistent global order to prevent compile-time cycle detection failures.

11. Per-CPU Variables

Native support for SMP architecture. Declaring a percpu variable compiles down to a %gs-relative access on x86-64. No address in the conventional sense exists.

percpu cpu_active_tasks: uint32

fn log_task() {
    percpu_write<uint32>(cpu_active_tasks, 0x01)
}

12. Extern Functions

Declares a function implemented in another compilation unit (C, assembly). Annotations must accurately describe the function's context and capabilities so the compiler can enforce call-edge rules across the boundary.

extern @ctx(thread, boot) @eff() @caps() fn platform_barrier();
extern @ctx(irq)           @eff() @caps() fn uart_irq_ack();
extern @ctx(thread, boot) @eff(block) @caps() fn sleep(uint64 us);

13. Advanced Kernel Primitives

Tail calls

A tail_call(callee, args) intrinsic discards the current stack frame before jumping. This guarantees zero stack growth, making it the only mathematically safe way to implement infinite polling loops inside fixed-stack @ctx(irq) handlers. tail_call requires --surface experimental.

@ctx(irq)
fn uart_poll(uint64 head) {
    uint64 next = (head + 1) & 7
    tail_call(uart_poll, next)   // jmp uart_poll — no frame growth
}

Critical blocks

A critical { } block declares an atomic critical section. The kernel profile (--profile kernel) forbids alloc, block, and yield effects anywhere inside a critical block.

fn write_char(uint8 b) {
    acquire(ConsoleLock)
    critical {
        UART0.Data = b     // no yield, no alloc inside
    }
    release(ConsoleLock)
}

Physical code shaping

Add @hotpath to a function and the compiler aligns its entry point to a 16-byte boundary, eliminating I-cache misses on tight interrupt dispatch loops.

Yield point

yieldpoint() inserts a voluntary CPU yield. Legal only in thread context, outside critical blocks, and in functions not marked @noyield.

@ctx(thread)
fn pump() {
    while pending() {
        process_one()
        yieldpoint()    // cooperatively yield to the scheduler
    }
}

14. Unsafe Blocks

An unsafe { } block enables operations that bypass the compiler's normal safety checks, such as inline assembly. All capability and context rules still apply inside an unsafe block.

fn flush_tlb() {
    unsafe {
        asm!(invlpg)
    }
}

Use unsafe only for unavoidable hardware interactions that have no safe surface equivalent.

15. Port I/O Intrinsics

KernRift provides built-in intrinsics for x86 port-mapped I/O. These emit native IN and OUT instructions directly — no extern fn declarations, no C FFI, no runtime overhead. Requires @module_caps(Ioport).

IntrinsicWidthx86 Instruction
inb(port) -> uint88-bitIN AL, DX
outb(port, val)8-bitOUT DX, AL
inw(port) -> uint1616-bitIN AX, DX
outw(port, val)16-bitOUT DX, AX
ind(port) -> uint3232-bitIN EAX, DX
outd(port, val)32-bitOUT DX, EAX
 
@module_caps(Ioport)

@ctx(thread, boot) @eff(ioport)
fn serial_write(uint8 b) {
    uint8 lsr = inb(0x3FD)
    while (lsr & 0x20) == 0 {
        lsr = inb(0x3FD)
    }
    outb(0x3F8, b)
}

x86_64 only. Compiling port I/O intrinsics with --arch arm64 produces a compile-time error — ARM has no port-mapped I/O.

16. Syscall Intrinsic

The @syscall intrinsic issues a raw system call to the host kernel. Available in @ctx(host) functions. Maps to syscall on x86_64 and svc on AArch64.

@ctx(host)
fn host_write(uint32 fd, uint64 buf, uint64 len) -> uint64 {
    uint64 result = @syscall(1, fd, buf, len)
    return result
}
PlatformInstructionNr register
Linux x86_64syscallRAX
Linux AArch64svc #0X8
macOS x86_64syscallRAX (nr + 0x2000000)
macOS AArch64svc #0x80X16

In practice, prefer the built-in host functions over raw @syscall unless you need a syscall not covered by the built-in set.

17. Built-in Host Functions

In @ctx(host) code, nine functions are available without extern fn declarations. The compiler maps these to __kr_* symbols provided by the KernRift host runtime.

FunctionDescription
write(fd, buf, len)Write len bytes from buf to file descriptor fd.
alloc(size) -> uint64Allocate size bytes. Returns a pointer.
dealloc(ptr, size)Free memory at ptr.
getenv(name) -> uint64Look up environment variable. Returns pointer or 0.
exec(cmd) -> uint32Execute a shell command. Returns exit code.
exit(code)Terminate the process.
str_copy(dst, src)Copy a null-terminated string.
str_cat(dst, src)Append src to the end of dst.
str_len(s) -> uint64Return the length of a null-terminated string.
 
@module_caps(Env, Process, Stdout)

@export
@ctx(host) @eff(env, process, stdout)
fn main() {
    uint64 msg = "Hello from KernRift host mode\n"
    write(1, msg, str_len(msg))

    uint32 rc = exec("ls -la")
    exit(rc)
}

These functions are only available in @ctx(host) code. Using them in kernel contexts (boot, thread, irq, nmi) is a compile-time error.

18. Slice Indexing

Slices support element access via bracket notation. Both reads and writes are supported. No bounds check is performed at runtime (freestanding environment).

Read

fn first_byte([uint8] data) -> uint8 {
    uint8 b = data[0]
    return b
}

Write

fn fill([uint8] buf, uint64 n, uint8 val) {
    uint64 i = 0
    while i < n {
        buf[i] = val
        i += 1
    }
}

The index expression can be any integer expression. data[i] loads the element at byte offset i * sizeof(T) from the slice base pointer.

19. Compiler CLI

Compile and run

$ kernriftc hello.kr                       # produces hello.krbo
$ kernrift hello.krbo                      # runs it, prints output to stdout
$ kernriftc --version

Emit binary artifacts

Use --emit=FORMAT to choose the output format. The default is a native-arch .krbo. krbofat produces a universal binary that runs on both x86-64 and ARM64.

# Fat binary (x86_64 + arm64 in one file)
$ kernriftc --emit=krbofat  -o out.krbo  driver.kr

# ELF relocatable object
$ kernriftc --emit=elfobj   -o out.o     driver.kr
$ kernriftc --emit=elfobj   --arch arm64 -o out.o driver.kr

# Textual assembly
$ kernriftc --emit=asm      -o out.s     driver.kr
$ kernriftc --emit=asm      --arch arm64 -o out.s driver.kr

# Static library archive (no external ar needed)
$ kernriftc --emit=staticlib -o out.a    driver.kr

# Native ELF executable (no external ld needed)
$ kernriftc --emit=elfexe    -o out      driver.kr

# Native host executable (no cc/gcc/clang needed)
$ kernriftc --emit=hostexe   -o build    build.kr

Check and analyse

$ kernriftc check driver.kr
$ kernriftc check --profile kernel driver.kr
$ kernriftc check --surface experimental driver.kr
$ kernriftc check --policy policy.toml driver.kr
$ kernriftc link file1.kr file2.kr      # cross-file lock-cycle check

Living compiler (auto-fix)

The living compiler analyses annotation spellings and can rewrite them to canonical form automatically.

$ kernriftc lc driver.kr                  # report non-canonical spellings
$ kernriftc lc --fix --dry-run driver.kr  # preview changes as a diff
$ kernriftc lc --fix --write driver.kr    # apply changes in-place
$ kernriftc lc --ci --min-fitness 70 driver.kr  # CI: fail below 70% fitness

Introspection

$ kernriftc --emit krir      driver.kr    # dump KRIR IR as JSON
$ kernriftc --emit lockgraph driver.kr    # dump lock graph
$ kernriftc --emit caps      driver.kr    # dump capability manifest
$ kernriftc --emit contracts driver.kr    # dump semantic contracts
$ kernriftc inspect-artifact out.krbo
$ kernriftc inspect-artifact out.krbo --format json

20. Binary Artifact Formats

.krbo — KernRift binary object

The native binary format. A single .krbo contains code for one architecture. 16-byte header layout:

OffsetSizeFieldDescription
04 BmagicKRBO (ASCII)
41 Barch0x01 = x86-64, 0x02 = AArch64
51 Babi0x01 = SysV
62 BreservedMust be zero
84 Bcode_lengthByte length of the code section
124 Bentry_offsetByte offset of the entry point within the code section

Fat binary

A fat binary begins with the 8-byte magic KRBOFATx and contains multiple architecture slices. kernrift automatically selects the matching slice at runtime. Recommended for distribution — a single file runs on both x86-64 and ARM64.

$ kernriftc --emit=krbofat -o firmware.krbo driver.kr
$ kernrift firmware.krbo    # auto-selects x86_64 or arm64 slice

Sidecar metadata

Add --meta-out to emit a JSON sidecar recording the artifact kind, surface profile, input path, byte size, and SHA-256 hash. Verify a binary against its sidecar at any time:

$ kernriftc --emit=krbo -o out.krbo --meta-out out.json driver.kr
$ kernriftc verify-artifact-meta out.krbo out.json

21. Contracts & Verification

KernRift emits strict JSON contracts that can be cryptographically hashed and signed using Ed25519. This prevents supply chain attacks and guarantees semantic integrity across builds.

Emitting and signing contracts

$ kernriftc check \
    --contracts-out contracts.json \
    --hash-out      contracts.sha256 \
    --sign-ed25519  secret.hex \
    --sig-out       contracts.sig \
    driver.kr

Verifying contracts

In a CI pipeline or loader, verify that the artifact exactly matches the signed JSON schema:

$ kernriftc verify \
    --contracts contracts.json \
    --hash      contracts.sha256 \
    --sig       contracts.sig \
    --pubkey    pubkey.hex

Policy evaluation

A policy file (TOML) encodes custom rules that are checked against the emitted contracts. Exit code 1 on any violation.

$ kernriftc check --policy policy.toml driver.kr
$ kernriftc policy --policy policy.toml --contracts contracts.json
$ kernriftc policy --evidence \
    --policy policy.toml \
    --contracts contracts.json
