# ARM64 Backend — Design Spec

## Overview

Add full AArch64 (ARM64) support to KernRift across three platforms: Linux (`aarch64-sysv`), macOS (`aarch64-macho`), and Windows (`aarch64-win`). The default `kernriftc` output becomes a fat binary (`KRBOFAT`) containing both x86_64 and ARM64 native code slices, compressed with LZ4. A new `--arch` flag allows targeting a single architecture. Scope: `crates/krir`, `crates/emit`, `crates/kernriftc`, `crates/kernrift`, `docs/spec`.

---

## 1. Phases

### Phase 1 — AArch64 Target Contracts

Define the three ABI target models in spec docs and in `crates/krir/src/lib.rs`:

- `aarch64-sysv` — Linux, AAPCS64
- `aarch64-macho` — macOS Apple Silicon, AAPCS64 + Apple stack rules
- `aarch64-win` — Windows ARM64

Each target gets a `BackendTargetContract` struct in `krir/src/lib.rs` (matching the shape of the existing x86_64 entries), a register map, and a stack contract. No instruction emission in this phase.

### Phase 2 — Instruction Selection + Register Allocation (Linux)

Add AArch64 instruction selection and register allocation for `aarch64-sysv`. Covers the same executable KRIR subset that x86_64 supports today: scalar ops, loads/stores, function calls, tail calls, MMIO. Produces a working ARM64 Linux binary. macOS pointer authentication (`PACIA`/`AUTIA`) is deferred to Phase 4.

### Phase 3 — Assembly + ELF Emission (Linux)

`--emit=asm` and `--emit=elfobj` for `aarch64-sysv`. Adds:
- `docs/spec/aarch64-asm-linear-subset-v0.1.md`
- `docs/spec/aarch64-object-linear-subset-v0.1.md`

### Phase 4 — macOS ARM64

Full stack for `aarch64-macho`: codegen (including pointer authentication), Mach-O object emission, assembly.

### Phase 5 — Windows ARM64

Full stack for `aarch64-win`: codegen, PE/COFF emission, assembly.

### Phase 6 — Fat Binary Container + CLI

Introduce the `KRBOFAT` container format, `--arch` flag, dual-file output for non-fat emits (`--emit=elfobj`, `--emit=asm`), and fat binary support in `kernrift`.

## 2. Fat Binary Container Format (`KRBOFAT`)

### Design principle

Single-arch krbo files remain byte-identical to the current format. The fat container is a new outer wrapper. A file is either a plain krbo or a `KRBOFAT` — distinguished by the first 8 bytes.

### File layout

```
[0..8]    magic:      "KRBOFAT\0"   (8 ASCII bytes + null terminator)
[8..12]   version:    u32 = 1       (little-endian throughout)
[12..16]  arch_count: u32
[16..]    entries:    arch_count × ArchEntry (32 bytes each)
          padding:    0-pad to next 8-byte boundary
          slices:     arch_count × LZ4-compressed krbo bytes
```

### `ArchEntry` (32 bytes)

```
arch_id:            u32   (1 = x86_64, 2 = arm64)
compression:        u32   (0 = none, 1 = lz4)
offset:             u64   (byte offset from file start to compressed slice)
compressed_size:    u64   (byte length of stored slice)
uncompressed_size:  u64   (byte length after decompression)
```

### Architecture IDs

| ID | Architecture |
|----|-------------|
| 1  | `x86_64`    |
| 2  | `arm64`     |

Reserved: IDs 3+ for future targets (RISC-V, etc.).

### Compression

Each slice is compressed independently with LZ4 (frame format). `compression = 0` is valid and means the slice is stored uncompressed — used when the slice is already small or when the user explicitly opts out. Default for fat binaries: `compression = 1` (LZ4).

**Dependency:** `lz4_flex` crate (pure Rust, no C bindings). Added to `crates/krir/Cargo.toml` or a new `crates/fat/Cargo.toml`.

### Size expectation

LZ4 typically compresses native code to 50–60% of original size. A fat binary containing both arches is therefore roughly **1.1–1.2×** the size of a single-arch krbo rather than 2×.

### Reading in `kernrift`

```
if file[0..8] == "KRBOFAT\0"  →  fat path: find ArchEntry for host arch,
                                            decompress slice, load
if file[0..8] == <krbo magic>  →  single-arch path (unchanged)
else                            →  error: unrecognised format
```

If the fat binary does not contain a slice for the host architecture:
```
error: <file> does not contain a slice for arm64
```

## 3. CLI Design

### `--arch` flag

Valid values: `x86_64`, `arm64` (also accept `aarch64` as alias for `arm64`).

```
kernriftc <file.kr>                               # fat krbo (default)
kernriftc --arch x86_64 <file.kr>                 # single-arch krbo, x86_64
kernriftc --arch arm64  <file.kr>                 # single-arch krbo, arm64
```

### Non-fat emit targets

`--emit=elfobj` and `--emit=asm` produce inherently single-arch output. Behaviour:

- **Without `--arch`**: produce two files, print a message.
- **With `--arch`**: produce one file (stem unchanged).

```
kernriftc --emit=elfobj <file.kr>
  → foo-x86_64.o + foo-arm64.o
  → emitted foo-x86_64.o and foo-arm64.o (use --arch to target one architecture)

kernriftc --arch arm64 --emit=elfobj <file.kr>
  → foo.o
```

Same pattern for `--emit=asm`.

### Arch-neutral emit targets

`--emit=krir`, `--emit=lockgraph`, `--emit=caps`, `--emit=contracts` are IR-level outputs — `--arch` is ignored for these.

### `kernrift` runner

No `--arch` flag. Always selects the host architecture automatically.

## 4. AArch64 Calling Conventions

### `aarch64-sysv` (Linux) and `aarch64-macho` (macOS) — AAPCS64

| Role | Registers |
|------|-----------|
| Integer args / return | `x0`–`x7` (return in `x0`) |
| Indirect result pointer | `x8` |
| Caller-saved (scratch) | `x9`–`x15` |
| Callee-saved | `x19`–`x28` |
| Frame pointer | `x29` |
| Link register | `x30` |
| Stack pointer | `sp` (16-byte aligned at calls) |

macOS-specific addition (Phase 4 only): pointer authentication on return addresses (`PACIA`/`AUTIA`). Scoped out of Phase 2.

### `aarch64-win` (Windows ARM64)

Same integer argument registers (`x0`–`x7`). Differences from AAPCS64:

- Stack alignment: 8-byte (not 16) at leaf calls, 16-byte at non-leaf calls
- `x18` is reserved as the TEB (Thread Environment Block) pointer — must never be used as a general-purpose register
- Otherwise the integer-scalar subset used by KernRift maps cleanly

### Tail calls on AArch64

x86_64 uses `jmp` for tail calls. AArch64 equivalent:
- Direct tail call: `b target` (branch without link)
- Indirect tail call: `br x16` (branch to register)

The tail-call constraint (callee must fit in the same register frame, ≤ 6 args under SysV) maps cleanly to AArch64.

- `tests/must_pass/arm64_hello.kr` — minimal function, compiles to fat krbo cleanly
- `tests/must_pass/arm64_mmio.kr` — device block + MMIO in AArch64 context
- `tests/must_pass/arm64_irq.kr` — `@ctx(irq)` function on AArch64

### Unit tests

- `fat.rs`: round-trip test — write fat binary with two mock slices, read back correct slice per arch ID
- `fat.rs`: LZ4 compress/decompress round-trip
- `fat.rs`: error path — host arch not present in fat binary

### Golden tests

- `lc` patterns still fire correctly on AArch64-compiled modules (telemetry is arch-neutral)
- `--emit=elfobj` without `--arch` produces two files with correct suffixes
- `--arch arm64 --emit=elfobj` produces a single `foo.o`
